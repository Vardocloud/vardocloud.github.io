# v10.6 — Speculative LLM + Zero Client Delay Architecture

**Date:** 2026-06-16 (same session as v10.4 → v10.5 → v10.6)
**Status:** CURRENT production
**Supersedes:** v10.5 (streaming + sentence TTS, still had 0.4s client-side delay)

## Key Innovations (Beyond v10.5)

### 1. Speculative LLM with Interim Results
Deepgram sends `interim_results` (partial transcriptions) BEFORE speech ends. v10.6 uses these to warm the LLM:

```python
# When interim transcript arrives (>15 chars):
speculative_task = asyncio.create_task(llm_stream(interim_text))
# LLM starts generating while user is still speaking

# When speech_final arrives with final transcript:
if speculative_task and not speculative_task.done():
    result = await speculative_task  # Already partially generated!
    # Use result if final text matches (usually does)
```

**Win:** LLM first-token latency drops from 2-5s → effectively 0s (already generating when speech ends).

### 2. Zero Client-Side Delay
No `delayed_flush()` timer. Process ONLY on Deepgram's `speech_final` or `is_final` events. Deepgram's own `utterance_end_ms=700` is sufficient silence detection.

### 3. Single-Sentence System Prompt
```
KRITIK KURAL: TEK CUMLE cevap ver. En fazla 15 kelime. Asla paragraf yazma.
```

Previous prompt said "2-3 cümle." Single-sentence responses mean:
- Faster TTS generation (1 sentence vs 2-3)
- Faster perceived turn-taking
- More natural conversational rhythm

### 4. Deepgram `utterance_end_ms=700`
Previous finding: minimum is 1000ms (500 causes HTTP 400).
v10.6 tests `700` — if it works, shaves 300ms off silence detection.
**Status:** Deployed, needs verification.

## Architecture Flow

```
User speaks → AudioContext PCM Int16 → WebSocket → Server → Deepgram WS
  ├─ interim transcript → speculative LLM starts immediately
  └─ speech_final with final text
       ├─ transcript sent to frontend (always)
       ├─ if speculative result ready → use it (saved 2-5s)
       ├─ else → start fresh LLM stream
       └─ sentences split by [.!?…]\s+
            └─ each sentence → TTS (ElevenLabs Bella, speed 1.1)
                 └─ audio sent to frontend immediately

If user speaks during playback:
  ├─ cancel_flag set → remaining sentences skipped
  ├─ current audio stopped in frontend (stopAudio())
  ├─ new speculative LLM starts with new interim text
  └─ new handle_reply launched
```

## Code Structure (v10.6)

```python
# Key functions:
connect_dg()        # Deepgram WS, utterance_end_ms=700
llm_stream(text)    # Speculative: streams LLM, returns full text
handle_reply(text)  # Uses speculative result, splits sentences, TTS each
process_dg()        # Routes interim → speculative, final → handle_reply
tts(sentence)       # Single-sentence TTS (speed 1.1, timeout 10s)
```

## Frontend Changes (v10.6)

- `AudioContext({sampleRate: 24000})` for playback (matching TTS output)
- `stopAudio()` function: stops AND disconnects current audio source
- `interrupt` message type: triggers `stopAudio()` + status update
- `assistant_text` type: incremental sentence display
- `assistant_full` type: single-sentence final display

## Known Issues / Open Questions

1. **Deepgram `utterance_end_ms=700`** — minimum was previously documented as 1000. If 700 causes 400 errors, revert to 1000.
2. **Speculative task cancellation** — if user speaks very quickly, multiple speculative tasks may stack. Current code cancels previous before starting new.
3. **Pollinations throttling** — consecutive requests get slower (5s →13s →20s). Speculative calls consume quota faster.
