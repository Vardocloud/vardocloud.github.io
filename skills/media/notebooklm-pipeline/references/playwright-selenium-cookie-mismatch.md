# Playwright → Selenium Cookie Format Mismatch (23 Haz 2026)

## Problem

NotebookLM MCP (notebooklm-mcp Python package) uses **Selenium** (undetected-chromedriver)
under the hood. The cookie export (`storage_state.json`) is generated by **Playwright**.
These two tools handle cookies differently, causing Google auth cookies to fail silently
during `_load_cookies()`.

## Symptoms

- Healthcheck returns `{status: "needs_auth", authenticated: false}`
- storage_state.json has valid Google auth cookies (expire months/years later)
- MCP Chrome process keeps crashing and respawning (multiple PIDs with different ports)
- Chrome profile path mismatch: some processes use `~/chrome_profile_notebooklm` (wrong)
  while others use `~/.hermes/chrome_profile_notebooklm` (correct)
- `chat_with_notebook` returns: "Not authenticated or browser not ready"

## Root Cause

The storage_state.json exported via Playwright's `page.context().storage_state()` includes:

1. **Cookies** — Playwright format with fields Selenium may not accept
2. **Origins** (localStorage + sessionStorage) — Selenium's `add_cookie()` CANNOT set these

### Cookie-level Issues

In `client.py`, `_load_cookies()` loops through cookies and calls `driver.add_cookie(c)`:

```python
for cookie in data.get("cookies", []):
    try:
        c = {
            "name": cookie["name"],
            "value": cookie["value"],
            "domain": cookie.get("domain", "").lstrip("."),
        }
        if cookie.get("path"): c["path"] = cookie["path"]
        if cookie.get("secure"): c["secure"] = cookie["secure"]
        if cookie.get("httpOnly"): c["httpOnly"] = cookie["httpOnly"]
        if cookie.get("sameSite"): c["sameSite"] = cookie["sameSite"]
        self.driver.add_cookie(c)
    except Exception:
        continue  # ← SILENTLY DROPS FAILED COOKIES
```

Problems with this approach:
- **Domain stripping:** `.lstrip(".")` removes ONLY the first dot from `.google.com`,
  but Google cookies use `.google.com` (wildcard subdomain). Selenium expects `google.com`.
  Strip works for simple cases but can fail for multi-dot domains like `.google.com.tr`
- **SameSite mismatch:** Playwright exports `sameSite: "Lax"` or `"Strict"` as strings.
  Selenium expects lowercase `"lax"` or `"strict"` — case sensitivity can cause rejection.
- **Expired cookies silently included:** The loop tries ALL cookies, including expired ones
  (like `__Secure-1PSIDRTS`). These also fail silently.
- **HttpOnly cookies:** Selenium's `add_cookie()` may reject HttpOnly cookies depending
  on browser state and domain readiness.
- **Secure cookies on HTTP:** If the page hasn't fully upgraded to HTTPS before cookie
  injection, Secure cookies are rejected.

### Origin-level Issues

Playwright's storage_state also captures `origins` (localStorage + sessionStorage per origin):

```python
"origins": [
    {
        "origin": "https://notebooklm.google.com",
        "localStorage": [{"name": "...", "value": "..."}]
    }
]
```

Selenium's `add_cookie()` ONLY handles cookies — localStorage and sessionStorage MUST be
set via JavaScript AFTER navigation, but `_load_cookies()` never does this.

Google may use localStorage flags to:
- Track authentication state beyond cookies
- Store anti-fingerprinting tokens
- Maintain session continuity signals

## Diagnosis Steps

### 1. Check which cookies failed

```python
python3 -c "
import json
data = json.load(open('/home/ubuntu/.notebooklm/profiles/default/storage_state.json'))
cookies = data.get('cookies', [])
from datetime import datetime, timezone
now = datetime.now(timezone.utc)
important = ['__Secure-1PSID','__Secure-3PSID','__Secure-1PSIDTS','__Secure-3PSIDTS',
             'SID','HSID','SSID','APISID','SAPISID','OSID']
for c in cookies:
    name = c.get('name','')
    for imp in important:
        if imp in name:
            expires = c.get('expires', 0)
            exp_date = datetime.fromtimestamp(expires, tz=timezone.utc) if expires else None
            status = 'VALID' if exp_date and exp_date > now else 'EXPIRED' if exp_date else 'session'
            print(f'{name:30s} | {status:10s}')
            break
"
```

If important cookies are `VALID` but MCP still says `needs_auth`, the issue is
cookie injection failure, not cookie expiry.

### 2. Check Chrome profile paths

```bash
ps aux | grep "chrome_profile_notebooklm" | grep -v grep | awk '{print $NF}' | sort -u
```

If you see TWO different paths (with and without the `.` dot prefix), zombie
Chrome instances from previous sessions are causing confusion:
- `--user-data-dir=/home/ubuntu/.hermes/chrome_profile_notebooklm` ✅ correct
- `--user-data-dir=/home/ubuntu/chrome_profile_notebooklm` ❌ stale/wrong

### 3. Check MCP healthcheck vs actual init

```bash
# healthcheck doesn't call _ensure_client — it only checks the flag
# If client is None → status: "unhealthy"
# If client exists but _is_authenticated=False → status: "needs_auth"
# If client exists and _is_authenticated=True → status: "healthy"
```

## Solutions

### Short-term: Re-export storage_state from the correct tool

If you used Playwright's `page.context().storage_state()`:

```python
# Playwright export format has 'cookies' + 'origins'
# The MCP's _load_cookies only handles 'cookies'
# Re-export using MCP-native cookie format or inject via SQLite
```

### Medium-term: Bypass Selenium's add_cookie, inject via SQLite

Directly write cookies to Chromium's SQLite Cookies database:

```bash
# 1. Stop Chrome that's using the profile
# 2. Open the Cookies DB
sqlite3 ~/.hermes/chrome_profile_notebooklm/Default/Cookies "
DELETE FROM cookies WHERE host_key LIKE '%google%';
"
# 3. Import cookies from storage_state via Python with sqlite3 module
```

### Long-term: Fix _load_cookies in client.py

Patches needed in `notebooklm_mcp/client.py`:

1. **Set localStorage after navigation:**
   ```python
   # After adding cookies, set localStorage from origins
   for origin_entry in data.get("origins", []):
       for item in origin_entry.get("localStorage", []):
           self.driver.execute_script(
               f"window.localStorage.setItem('{item['name']}', '{item['value']}')"
           )
   ```

2. **Fix SameSite casing:**
   ```python
   if cookie.get("sameSite"):
       c["sameSite"] = cookie["sameSite"].lower()
   ```

3. **Skip expired cookies explicitly:**
   ```python
   import time
   expires = cookie.get("expires", 0)
   if expires and expires < time.time():
       continue
   ```

4. **Visit the domain BEFORE adding cookies with explicit HTTPS:**
   ```python
   self.driver.get("https://notebooklm.google.com")
   # Wait for full HTTPS connection before adding secure cookies
   ```

### 🧹 Cleanest approach: Empty storage_state + Playwright persistent context (23 Haz 2026)

The root problem is that MCP's `_load_cookies()` runs EVERY time the MCP starts,
and it tries to inject Playwright cookies into Selenium. Even if injection
*fails silently*, the attempt itself doesn't break anything. But if any cookies
are partially injected, they can corrupt the working profile session.

**The fix:** After seeding the profile with a valid session:

1. **Overwrite storage_state.json with empty data:**
   ```python
   with open(STORAGE_PATH, "w") as f:
       json.dump({"cookies": [], "origins": []}, f)
   ```
2. **Start MCP.** `_load_cookies()` loads 0 cookies (no-op). 
3. `_authenticate_sync()` navigates to NotebookLM. Since the Chrome profile
   (via `--user-data-dir`) already has the session from Playwright's persistent
   context, Google recognizes the session and grants access.
4. MCP sets `_is_authenticated = True` and reports `healthy`.

**Why this works:** The profile on disk (SQLite Cookies DB, Local Storage) is
the SOURCE OF TRUTH, NOT storage_state.json. storage_state.json is just a
portable snapshot for Playwright. The MCP loads it because it doesn't know any
better. By emptying it, we force the MCP to rely on the actual profile, which
already has everything it needs.

### ⚠️ Google Bot Protection Exhaustion (23 Haz 2026)

After ~5-10 automated login attempts from the same IP address within a short
window, Google enters a **persistent "signin/rejected" state** that is almost
impossible to recover from automatically:

```
https://accounts.google.com/v3/signin/rejected?continue=...
```

**Characteristics:**
- ALL subsequent attempts fail — headed, headless, Playwright, CDP, different Chrome flags
- The page shows "Couldn't sign you in" or "This browser or app may not be secure"
- Container/Docker restarts do NOT reset this — it's bound to the IP address
- Only 3 cookies saved (compared to 50+ when logged in)
- No visible captcha or challenge — just a blank or redirect page

**Recovery options:**
1. **Wait 30+ minutes** — the IP-based block eventually expires (exact time unknown)
2. **Use stored cookies** from a previously successful login (before the block started)
3. **Switch IP** — change proxy or use a different egress point
4. **VNC manual login** — even manual login via a real browser on the same IP may still
   be rejected until the cooldown expires

**Prevention:**
- Limit automated login attempts to MAX 3 before switching to manual (VNC) flow
- Cache successful cookie state and reuse it — don't attempt fresh login every time
- Use a single persistent Chrome profile instead of creating new profiles per attempt
- If the FIRST attempt shows `signin/rejected`, stop immediately and use VNC

### Nuclear option: Switch to Playwright-based auth

The MCP uses Selenium because of undetected-chromedriver for anti-detection.
But Playwright also has anti-detection capabilities. Consider switching the
MCP's internal driver to Playwright (via playwright-python) which natively
handles `storage_state.json` through `page.context().add_cookies()` and
`page.context().add_init_script()` for localStorage.

### Practical fix: Playwright persistent context to seed MCP profile (23 Haz 2026)

Instead of fixing the MCP's internal code, seed the Chrome profile BEFORE
starting the MCP. This works because the MCP uses `--user-data-dir=<profile>`
with `use_persistent_session=True`:

```python
from playwright.async_api import async_playwright

async with async_playwright() as pw:
    browser = await pw.chromium.launch_persistent_context(
        MCP_PROFILE_PATH,  # same as MCP's --user-data-dir
        headless=True,  # or False for VNC visibility
        args=["--no-sandbox", "--disable-dev-shm-usage"],
    )
    page = browser.pages[0]
    await page.goto(NOTEBOOK_URL)
    
    # Complete the login flow (automated or manual)...
    # The profile is now seeded with valid cookies on disk
    
    await browser.close()
    # Profile persists! Start MCP now — it will reuse this profile.
```

**Why this works:** `launch_persistent_context` writes cookies directly to
the Chrome profile's SQLite Cookies database on disk. When MCP's
undetected-chromedriver starts later with `--user-data-dir=<same_profile>`,
the cookies are already there — no injection needed.

**⚠️ Caveat:** If the automated login triggers Google bot protection
(`signin/rejected`), you must use `headless=False` + VNC for manual login.
The persistent context approach only avoids the Selenium injection problem —
it does NOT bypass Google's bot detection.

**Try Another Way automated flow (works with fresh profile):**
```python
# After page.goto(NOTEBOOK_URL) shows accounts.google.com/login:
for attempt in range(10):
    url = page.url
    if "notebook" in url and "accounts" not in url:
        break  # logged in!
    
    # 1. Account chooser
    acct = await page.query_selector(f'[data-identifier="{email}"]')
    if acct: await acct.click(); await page.wait_for_timeout(5000); continue
    
    # 2. Email input  
    email_el = await page.query_selector("#identifierId")
    if email_el: await email_el.fill(email); await page.keyboard.press("Enter"); continue
    
    # 3. Password (visible only)
    pw = await page.query_selector('input[type="password"]')
    if pw and await pw.is_visible(): await pw.fill(passwd); await page.keyboard.press("Enter"); continue
    
    # 4. Try Another Way (bypasses passkey)
    taw = await page.query_selector('a:has-text("Try another way")')
    if taw and await taw.is_visible(): await taw.click(); continue
    
    # 5. Enter your password option
    epo = await page.query_selector('[data-value="PASSWORD"]')
    if epo: await epo.click(); continue
```

**Important:** The "Try Another Way" → "Enter your password" flow works
RELIABLY with Playwright (unlike Selenium's `.click()` which React ignores).
Use `page.query_selector` + `is_visible()` checks. Password field appears
only after "Enter your password" is selected from the passkey bypass menu.
